Internals of Client-Side DB Sharding

Problem Statement

Scaling a write-heavy application with a single relational database often leads to bottlenecks:

  • Maxed-out connection pools
  • High write contention
  • Large, unwieldy tables

Sharding — splitting data across databases — is the long-term fix. But setting it up with Hibernate and Dropwizard is error-prone and complex.

Scaling with Application-Level Sharding

As applications grow, especially those that are write-heavy, traditional relational databases begin to show limitations:

  • Connection pool limits get saturated
  • Indexes grow large, affecting performance
  • Write locks and ACID constraints create contention
  • Replication helps reads, but not writes

This is where sharding comes in — splitting data horizontally across multiple databases.

What is Application-Level Sharding?

Application-level (aka client-side) sharding means the application itself contains the logic to decide which database shard to route a query to. This gives complete control to the app, without relying on a central router or proxy.

Typical flow:

  • App receives request with a sharding key (e.g., userId)
  • App hashes key → gets a logical bucket → maps to a shard
  • Query is routed directly to the correct database instance

Real-World Examples

  • Notion: Client-side routed Postgres shards using document ID as key
  • PayPal: Uses sharded customer databases with routing logic embedded in services
  • Netflix: Embeds shard selection logic within microservices for high-throughput use cases

Enter Dropwizard's DB Sharding Bundle

The dropwizard-db-sharding-bundle is a powerful abstraction for implementing application-level sharding in Java services using Hibernate and Dropwizard. It simplifies:

  • SessionFactory management per shard
  • Routing logic based on sharding key
  • DAO design for top-level (LookupDao) and relational (RelationalDao) entities

Rather than writing boilerplate for shard resolution and session handling, the bundle gives you:

  • 1024 logical buckets mapped via Murmur3 hashing
  • ShardCalculator + ShardManager to map key → shard
  • Pluggable strategies (Balanced, Legacy, Multi-Tenant)

This brings the power of sharding into your app without overwhelming it with complexity.

When to Use It

Application-level sharding via this bundle is great when you:

  • Have a write-heavy workload and a known shard key (like userId)
  • Want to scale horizontally without changing your ORM logic
  • Need something production-ready, yet lightweight and transparent

As your system grows, you may graduate to service-side sharding, but this bundle provides a rock-solid foundation to get started.

Sharding Strategy: Client-side vs Service-side

There are multiple ways to implement sharding in large-scale systems. The dropwizard-db-sharding-bundle uses client-side sharding, but let's explore both common approaches.

1. Client-side Sharding (Used by Dropwizard Bundle)

  • Shard routing logic lives inside the application code
  • Each app embeds the shard calculation + routing (via ShardCalculator + ShardManager)
  • No extra network hops — routing is local and fast

Drawbacks: Difficult to evolve routing rules or support dynamic shards without app redeploys.

2. Service-side Sharding

  • A dedicated "Shard Router" service determines which shard to use
  • Clients query the service or it acts as a proxy
  • Allows centralized control and coordination

Drawbacks: Adds an extra network hop and infrastructure complexity.

The Solution: dropwizard-db-sharding-bundle

This Dropwizard bundle abstracts all the pain away. It provides:

  • Shard definition and connection setup in config.yaml
  • Separate Hibernate SessionFactory for each shard
  • Sharding-aware DAOs with automatic routing
  • Multi-tenant and legacy routing support

Lifecycle & Initialization Flow

  1. Bundle reads shard definitions from YAML
  2. Initializes multiple SessionFactory instances
  3. Creates a ShardManager and a ShardCalculator
  4. Registers DAOs and wraps them in MultiTenantDelegates
  5. Health checks and admin tasks (e.g., shard blacklisting) are registered

Shard Key Mechanics

The bundle uses Murmur3 hash on a logical key (e.g. userId) to assign one of 1024 logical buckets. These buckets are mapped to physical shards by the ShardManager.

@LookupKey
private String userId; // Key used to hash and find the shard

Sample: LookupDao

LookupDao<User> dao = shardingBundle.createParentObjectDao(User.class);
User user = new User("user123", ...);
dao.save(user); // Routed to correct shard

Sample: RelationalDao

RelationalDao<Order> dao = shardingBundle.createRelatedObjectDao(Order.class);
dao.save("user123", new Order(...));

Internals & Routing Logic

The actual routing occurs inside MultiTenantLookupDao and MultiTenantRelationalDao:

  1. Uses reflection to read the @LookupKey
  2. Computes bucket via ShardCalculator
  3. Maps bucket to shard via ShardManager
  4. Delegates the Hibernate call to the correct DAO

Advanced Capabilities

  • Cacheable DAOs: Pluggable local cache layer
  • Shard Blacklisting: Temporarily ignore failing shards
  • Custom Calculators: Implement your own hashing logic
  • Legacy Mode: Modulo routing (instead of consistent hashing)

Learnings

  • Uses delegate pattern internally
  • Highly extensible via generic interfaces
  • Session management fully abstracted
  • Cleanly separates business logic from infra logic
